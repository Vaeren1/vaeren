## 1. Architektur-Entscheidungen (autonom)

| Entscheidung | Wahl | Begründung |
|---|---|---|
| **Modul-Struktur** | Eigene Django-App `iso27001/` als Tenant-Schema-App (in `TENANT_APPS` in `backend/config/settings/base.py`). | Spec §1 Domain-driven boundaries. Analog zu `ki_inventar`/`nis2`/`datenpannen`. |
| **Stammdaten-Sitz** | `Iso27001Control` lebt im Tenant-Schema (nicht Public), wird pro Tenant via Daten-Migration geseedet. | Vereinfacht Queries (kein Cross-Schema-Join). Speicher-Overhead = 93 Rows × Tenant-Anzahl, akzeptabel. |
| **Seed-Quelle** | Statische JSON-Datei `iso27001/data/annex_a_2022.json`, geladen via Daten-Migration `0002_seed_annex_a` + Management-Command `seed_iso27001_controls`. | Versionierbar im Repo, einfach lesbar/updatebar. ISO-Norm-Text ist nicht urheberrechtlich geschützt (Reproduktion der Control-Codes + dt. Übersetzung in eigenen Worten). |
| **Encryption** | Keine. ISO-Daten müssen für Auditor lesbar bleiben. | Spec §11. AuditLog statt Encryption als Manipulationsschutz. |
| **Polymorphic-Pattern** | `IsoTask(ComplianceTask)` mit Multi-FK-Pattern (implementation/risiko/audit/finding/mgt_review), genau eine nicht-null. | Konsistent mit `DatenpannenTask`, `MeldungsTask`. |
| **Auto-Mapping-Implementierung** | Statische Hash-Tabelle in `iso27001/mapping.py` (Modul → Control-Codes-Liste) + pro Modul eine `suggest_*`-Funktion. | YAGNI gegenüber Engine. Mensch bestätigt jeden Vorschlag — Auto-Apply verboten. |
| **LLM-Aufruf** | Über `core.llm_client.generate()`, gleiches Pattern wie `ki_inventar/llm.py`. Static-Fallback-JSON definiert. | Recycelt bestehende Infrastruktur inkl. RDG-Layer-2. |
| **PDF-Render** | WeasyPrint, gleiches Pattern wie Pflichtunterweisung-Zertifikate. Templates in `iso27001/templates/iso27001/*.html`. | Bereits installiert + im Produktiv-Stack erprobt. |
| **SoA-Versionierung** | Snapshot in `StatementOfApplicability.snapshot_data` (JSONField), nicht eine Tabelle pro Version. | Auditor-Pflicht: Versions-Diff später möglich, aber jetzt YAGNI. |
| **Frontend-State** | TanStack Query für API, kein Zustand für Modul-State (nur Shared App-State). | Konsistent mit anderen Modulen. |
| **Permission-Modell** | `iso27001.lesen` für GF + CB + QM; `iso27001.bearbeiten` nur für CB + QM. Management-Review-Genehmigung (`mgt_review.genehmigen`) nur GF. | Spec §6 (MVP-Architektur) Tabelle. |
| **Auditor-Rolle** | NICHT in Phase 3. Externe Auditor-Lesezugriffe kommen ab erstem realen Audit-Bedarf (Phase 4). | YAGNI bis Pilot. |

## 2. Schritt-für-Schritt-Plan

### Schritt 1 — App-Skelett + Settings (1 h)
- Neue Django-App `backend/iso27001/` mit `apps.py`, `__init__.py`, `models.py`, `serializers.py`, `views.py`, `urls.py`, `signals.py`, `llm.py`, `mapping.py`, `permissions.py`, `data/annex_a_2022.json`, `templates/iso27001/`, `tests/__init__.py`.
- Eintrag in `backend/config/settings/base.py::TENANT_APPS` ergänzen: `"iso27001"`.
- URL-Include in `backend/config/urls.py` (oder Tenant-URLs-File): `path("api/iso27001/", include("iso27001.urls"))`.
- `iso27001/apps.py::ready()` registriert `signals`.
- Smoke-Test: `python manage.py check` läuft sauber.

### Schritt 2 — Seed-Daten + Katalog-Model (3 h)
- `iso27001/models.py`: `Iso27001Control` mit `code`, `name`, `description_de`, `kategorie`, `applicability_default`, `iso_clause`, `sortier_index` (siehe Spec §4.2).
- `iso27001/data/annex_a_2022.json`: Liste mit 93 Einträgen, kategorisiert in A.5 (37), A.6 (8), A.7 (14), A.8 (34). Inhalt: dt. Kurz-Beschreibungen (eigene Worte, keine ISO-Norm-Zitate).
- `iso27001/management/commands/seed_iso27001_controls.py`: Liest JSON, ruft `update_or_create` pro Code. Idempotent.
- Migration `0001_initial.py` (Schema) + `0002_seed_annex_a.py` (RunPython-Migration → ruft `seed_iso27001_controls`-Logik auf für tenant-Schema).
- **Test:** `iso27001/tests/test_seed.py` mit zwei Tests: (a) nach `seed` existieren 93 Controls, (b) zweiter Run ändert nichts (idempotent).

### Schritt 3 — Implementation-Model + Evidence-Link-Model (2 h)
- `iso27001/models.py`: `ControlImplementation` + `ControlEvidenceLink` + `ImplementationStatus`-Choices (Spec §4.2).
- Migration `0003_implementation_and_evidencelink.py`.
- `iso27001/serializers.py`: `ControlListSerializer`, `ControlDetailSerializer`, `ControlImplementationSerializer`, `EvidenceLinkSerializer`.
- **Test:** `iso27001/tests/test_models.py` — Constraints, unique_together, FK-Cascade-Verhalten, Default-Werte.

### Schritt 4 — Mapping-Engine + Auto-Suggestion (2,5 h)
- `iso27001/mapping.py`: 
  - Statisches Dict `MODULE_TO_CONTROLS: dict[str, list[str]]` (siehe Spec §4.4, ≥30 Mappings).
  - Funktionen `suggest_from_ki_inventar`, `suggest_from_nis2`, `suggest_from_avv`, `suggest_from_datenpannen`, `suggest_from_hinschg`, `suggest_from_pflichtunterweisung`. Jede returnt `list[Evidence]`.
  - Master-Funktion `suggest_evidence_links_for(implementation: ControlImplementation) -> list[Evidence]`.
- `iso27001/signals.py`: `post_save` auf `ControlImplementation` (created=True) → ruft `suggest_evidence_links_for` + erstellt `ControlEvidenceLink(auto_suggested=True, confirmed_by=null)`.
- **Test:** `iso27001/tests/test_mapping.py` — pro 6 Quell-Module mind. 1 erwarteter Mapping-Treffer; `test_signals.py` — nach `ControlImplementation.objects.create(...)` existieren ≥1 ControlEvidenceLink wenn Demo-Daten der Quellen vorhanden sind.

### Schritt 5 — Controls-API (2 h)
- `iso27001/views.py`: `ControlViewSet` (ReadOnly, listet alle 93 Controls + Joined-Implementation), `ControlImplementationViewSet` (Patch + custom Actions `verify`, `llm_entwurf`, `evidence_suggestions`).
- `iso27001/urls.py`: Router-Registrierung.
- `iso27001/permissions.py`: `Iso27001Permission(RulesPermission)` mit `view_rule = "can_view_iso27001"`, `edit_rule = "can_edit_iso27001"`.
- `core/rules.py`: 2 neue Predicates `can_view_iso27001`, `can_edit_iso27001` (analog vorhandenes Pattern).
- OpenAPI-Schema generieren + commiten (`make schema`).
- **Test:** `iso27001/tests/test_api_controls.py` — DRF-APIClient-Tests für List + Detail + Patch + Verify. `test_permissions.py` — 5 Rollen × 3 Actions = 15 Permission-Assertions.

### Schritt 6 — LLM-Entwurfshilfe + RDG-Validator (3 h)
- `iso27001/llm.py`: 
  - `entwurf_implementation_beschreibung(control: Iso27001Control, tenant_kontext: dict) -> ImplVorschlag` analog `klassifiziere_ki_tool`.
  - `entwurf_treatment_plan(risiko: IsmsRiskAssessment) -> TreatmentVorschlag`.
  - System-Prompt mit Konjunktiv + „Entwurf:"-Pflicht, Verbot konformitäts-bestätigender Phrasen.
  - Static-Fallback-JSONs.
- `core/llm_validator.py`: `ISO_FORBIDDEN_PHRASES` ergänzen + in der zentralen Validator-Funktion einbinden.
- View `POST /api/iso27001/implementations/{id}/llm-entwurf/`: Ruft LLM, schreibt Ergebnis in `implementation_vorschlag` (NICHT in `implementation_beschreibung`), gibt Vorschlag + Disclaimer zurück.
- **Test:** `iso27001/tests/test_rdg_validator.py` — Mock-LLM gibt Output mit verbotener Phrase → Validator schlägt an, Fallback aktiviert. Mock-LLM gibt sauberen Output → Vorschlag wird übernommen.

## 4. Migrations-Reihenfolge

1. `iso27001/migrations/0001_initial.py` — Iso27001Control + indexes.
2. `iso27001/migrations/0002_seed_annex_a.py` — Daten-Migration mit RunPython, lädt JSON.
3. `iso27001/migrations/0003_implementation_und_evidence.py` — ControlImplementation + ControlEvidenceLink.
4. `iso27001/migrations/0004_risk_und_asset.py` — IsmsAsset + IsmsRiskAssessment.
5. `iso27001/migrations/0005_soa.py` — StatementOfApplicability.
6. `iso27001/migrations/0006_mgt_review_und_audit.py` — ManagementReview + InternesAudit + AuditFinding + IsoTask.

Alle Migrationen müssen rückwärtskompatibel sein (CLAUDE.md §Deployment). Reverse-Migration darf Daten nicht verlieren — `RunPython` mit `reverse_code=migrations.RunPython.noop` für Seed-Migration, sonst Daten-Verlust beim Rollback (akzeptabel im Erst-Deploy-Fall).

`python manage.py migrate_schemas --noinput` läuft im Container-CMD vor Daphne-Start — Reihenfolge automatisch.

## 5. Risiko-Mitigation

| Risiko | Mitigation |
|---|---|
| **93er-Seed-Datenpflege** | JSON-File ist Single-Source-of-Truth, Test stellt Existenz aller 93 Einträge sicher. Update bei Norm-Revision via PR auf JSON-File. |
| **WeasyPrint-PDF-Render-Memory-Spike bei 93 Controls** | Test mit fullem Seed-Set; falls >2 s Render-Zeit, in Phase 4 auf Celery-Task umstellen. |
| **LLM-Entwurf füllt versehentlich `implementation_beschreibung`** | Serializer-Klasse trennt strikt: `implementation_vorschlag` ist read-only von außen außer über die `llm-entwurf`-Action; `implementation_beschreibung` ist via Patch editierbar. Tests in `test_api_controls.py` verifizieren. |
| **Auto-Mapping erzeugt zu viele falsche Vorschläge** | `auto_suggested=True`-Eintrag braucht explizite Confirmation via `confirm/`-Endpoint. UI zeigt unbestätigte Vorschläge in eigener Section mit „Vorschlag — bitte prüfen"-Badge. |
| **Tenant ohne Quell-Daten sieht leeres Auto-Mapping** | Akzeptabel — Mensch füllt manuell via Evidence-Upload. Onboarding-Hinweis im Dashboard („Aktiviere zuerst AVV-Modul für vollständige Evidence-Coverage"). |
| **Performance-Probleme bei Coverage-Score-Berechnung** | Score-Berechnung sind 93 Rows × 1 Status-Aggregation = trivial. Single-Query mit `aggregate()`. |
| **Cross-Tenant-Datenleck** | Multi-Tenant-Isolation-Test in CI-Gate (Schritt 14). Tenant-Schema-Isolation reicht. |
| **Feature-Scope explodiert** | Strenger Cut: Mgt-Review-Auto-Vorbefüllung darf maximal Daten der 6 bestehenden Module aggregieren — KEIN neuer Input-Typ. Audit-Findings-Modul OHNE Sub-Workflow-Engine (CAPA, Wirksamkeits-Reife-Stufen etc.). |
| **PDF-Templates brauchen Design-Review** | Erste Version Roh-Tabellen, kein Logo-Branding. Design-Pass in Phase 3b möglich. |
